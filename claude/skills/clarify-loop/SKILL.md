# Clarify Loop

## 目的

透過結構化的迭代訪談，幫助使用者和 stakeholder 釐清需求，
每輪產出清楚的需求摘要，並可持續帶入新的 spec 進行下一輪澄清。

## 執行流程

### 第一步：建立 Session 記錄

Session 開始時，在 `~/.clarify-sessions/` 建立新檔案，檔名為 `YYYY-MM-DD-HHMM.md`，
立即將使用者的原始 idea **逐字複製**（不摘要、不改寫）寫入檔頭：

```markdown
# Clarify Session YYYY-MM-DD HH:MM

## Idea

<使用者原始 idea 全文>

## Q&A

```

每題回答採納後，append 一行：
```
- Q: <問題> → A: <最終答案>
```

### 第二步：識別資訊缺口（內部評估，不輸出）

閱讀使用者提供的問題或 spec，評估以下面向，找出最模糊的 3–5 個：

- **Why** — 目標是什麼？為什麼要做這件事？
- **What** — 範圍是什麼？包含什麼、不包含什麼？
- **Who** — 使用對象是誰？誰是主要 stakeholder？
- **How** — 有技術、流程、或架構上的限制嗎？
- **Done** — 什麼情況算完成？成功標準是什麼？

可以直接推斷的內容不要問，直接寫入摘要。從影響最大的缺口開始提問。

### 第三步：一次一題，最多五題

每次只問一個問題，使用以下格式：

---
**[Q N/5]** [問題內容，具體描述你想了解的決策點]

**推薦：X** — [1–2 句推薦理由，說明為什麼這個選項最常見或最合理]

| 選項 | 說明 |
|------|------|
| A | [選項描述] |
| B | [選項描述] |
| C | [選項描述] |
| D | 其他（請描述） |

回覆選項代號即可，或說「yes」接受推薦。

---

等使用者回答後：
- 說 "yes" / "推薦" → 採用推薦選項
- 選某個選項 → 採用該選項
- 回答模糊 → 追問釐清（**不計入題數**）

採納後立即 append 紀錄，然後直接進入下一題，不展示更新內容。

**問題設計原則：**
- 每題針對一個決策點，不混問
- 推薦要真的有理由，不是隨便選一個
- 選項涵蓋常見情境（讓使用者不用從零想）
- 選項數量 3–4 個，加上「其他」
- 選項之間盡量互斥
- 若選項無意義（如填寫名稱、日期），改用簡答題格式：

```
**[Q N/5]** [問題內容]

**建議：** [你的建議答案] — [理由]

請簡短回答，或說「yes」接受建議。
```

### 第四步：判斷停止時機

滿足以下任一條件即可停止提問：
- 已問滿 5 題
- 核心目標、使用情境、關鍵限制、成功標準都已清楚

不要過度訪談。資訊夠用就停。

### 第五步：產出需求摘要

每輪結束後，輸出以下格式的摘要（固定中文）：

---
## 需求摘要（第 N 輪）

**背景**
[1–2 句描述問題情境]

**目標**
[具體描述要達成什麼]

**使用對象**
[誰會用到、誰受影響]

**範圍**
- 包含：…
- 不包含：…

**關鍵需求**
1. …
2. …

**限制條件**
- …（無則省略）

**成功標準**
- [ ] [可驗證的條件]
- [ ] [可驗證的條件]

**待確認事項**（如有）
- …

---

### 第六步：詢問是否繼續下一輪

> 這輪澄清完成，紀錄已儲存。如果有新的 spec 或想進一步細化某個面向，請貼上內容，我們繼續下一輪。

若使用者貼入新的 spec 或說「繼續」，則從第二步重新開始，輪數 +1。

## 多輪迭代說明

- 每輪都是獨立的五題額度
- 新的一輪可以聚焦在上一輪的「待確認事項」
- 摘要標題標記輪數（第 1 輪、第 2 輪…）方便追蹤演進
- 若使用者說「沒了」或「結束」，輸出所有輪次的摘要合併版本作為最終文件
