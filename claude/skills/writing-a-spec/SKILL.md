# 撰寫 spec

把一個功能想法，透過澄清問答收斂成一份**標準 spec**，並寫入 `docs/<domain>/<feature>/spec.md`。

核心理念：好的 spec 不是憑空寫出來的，是「問對問題」問出來的。所以這個 skill 先補齊資訊缺口，再落地成結構化文件——而不是拿著模糊需求硬寫、留下一堆錯誤假設。

## 範圍：這是 spec（WHAT／WHY），不是 plan（HOW）

這份文件回答**要解決誰的什麼問題、為什麼值得做、做到什麼程度算成功**——也就是 WHAT 與 WHY。
它**不**回答 HOW：不寫系統架構、資料模型、API 契約、類別／函式設計、或實作步驟。那些屬於之後另寫的 plan（實作規劃文件 / design doc）。

為什麼要守住這條線：spec 的讀者是跨職能的人（產品、設計、工程、主管），他們得先對「做什麼、為何做」達成共識，工程才接手決定「怎麼做」。把實作細節混進來，會讓文件過早綁死方案，也讓非工程讀者讀不下去。沒有約束時，模型很容易自己長出「API 設計」「資料模型」這類章節——這就是要主動擋掉的漂移。

實務界線：

- 當需求或你自己的推理開始滑向實作（「用哪個資料庫」「拆哪些 service」「API 長怎樣」），**停下來**：要嘛省略，要嘛在「開放問題」留一句指標（例：「實作方式待 plan 決定」），不要長成設計章節。
- 「非功能需求」用**結果**陳述、不是**方案**：寫「需在 200ms 內回應」「需支援中英文」，而不是「用 Redis 快取」「接 Elasticsearch」。前者是產品要求，後者是實作選擇。

## 語言

spec 內容與章節標題，跟著使用者當下溝通的語言走（預設繁體中文；若使用者以英文溝通，整份 spec 改用英文）。

## 執行流程

### 第一步：盤點已知

先讀懂使用者的需求描述，並善用手邊脈絡（現有程式碼、相關文件、對話歷史）。
能直接推斷的就推斷、寫入，**不要明知故問**。把「還缺什麼才寫得出完整 spec」想清楚——這些缺口就是下一步要問的。

對照 spec 必要章節，檢查哪些面向資訊不足：

- **問題 / Why** — 要解決什麼痛點？為什麼現在做？
- **目標 / Success** — 怎樣算成功？有可量測指標嗎？
- **使用者 / Who** — 誰會用？主要情境是什麼？
- **範圍 / Scope** — 包含什麼、明確不做什麼？
- **功能 / What** — 核心功能行為有哪些？
- **限制 / Constraints** — 效能、安全、相容性等非功能要求？
- **驗收 / Done** — 怎麼驗證完成？

### 第二步：澄清缺口（最多 5 題）

一次只問一題，讓使用者好回答。格式：

---
**[Q N/5]** [針對一個決策點的具體問題]

**推薦：X** — [1–2 句推薦理由]

| 選項 | 說明 |
|------|------|
| A | … |
| B | … |
| C | … |
| D | 其他（請描述） |

回覆代號即可，或說「yes」接受推薦。

---

若該題不適合用選項（如命名、數值），改用簡答：給出建議 + 理由，請使用者確認。

提問原則：

- 每題只針對一個決策點，不混問
- 從影響最大的缺口問起
- 推薦要有真實理由，不是隨便挑一個
- 回答模糊時追問釐清（**不計入題數**）

**何時停**：問滿 5 題，或核心目標／使用情境／關鍵限制／驗收標準都清楚了——資訊夠用就停，不要過度訪談。

**降級路徑**：若使用者明確說「直接寫／不用問」，或當下無法互動取得答案（例如自動化執行），就跳過提問，以合理假設直接產出 spec，把假設與未定項清楚標在 `[待確認]` 並列入「開放問題」。誠實標記未知，比卡住不產出有用。

### 第三步：確認檔案路徑

依需求推斷 `domain`（所屬領域，如 billing、auth、search）與 `feature`（具體功能，如 invoice-export），兩者皆用 kebab-case。組成路徑後**先回報給使用者確認再寫檔**：

> 我會把 spec 寫到 `docs/billing/invoice-export/spec.md`，domain／feature 命名 OK 嗎？

若使用者已在需求中明確指定 domain／feature，直接採用、不用再問。
若 `spec.md` 已存在，先讀它，並問使用者要「更新既有內容」還是「另開新檔」，不要直接覆蓋既有規格。

### 第四步：撰寫 spec

讀 `assets/spec-template.md` 作為骨架，逐節填寫：

- 守住 spec 邊界：只寫 WHAT／WHY，不寫架構／資料模型／API／實作細節（見上方「範圍」）
- 每節都要處理：沒有內容的章節用一句話說明「本功能不適用」，而非留白
- 經澄清仍未定的事，寫進「開放問題」並在對應章節標 **[待確認]**，**絕不自己編造**
- 功能需求編號（FR-1、FR-2…）並標優先級（P0／P1／P2），讓驗收標準能逐條對應
- frontmatter 的 `created`／`updated` 填今天日期，`status` 預設 `draft`

### 第五步：寫檔並回報

建立目錄、寫入檔案，然後簡短回報：

- 檔案路徑
- 一句話摘要這份 spec 涵蓋什麼
- 哪些章節已完整、哪些還留有 `[待確認]`
- 建議下一步（例如：補齊開放問題、或進入實作規劃）

## 為什麼這樣設計

- **先問再寫**：拿模糊需求硬寫出來的 spec 會充滿錯誤假設，事後改它比一開始問清楚更貴。
- **固定範本**：統一結構讓 reviewer 一眼找到關鍵資訊，也讓不同 spec 之間可比較、可累積。
- **標 `[待確認]` 而非編造**：誠實標記未知，比用看似合理的猜測掩蓋缺口安全——猜測會在實作期才爆炸，且難追溯。
